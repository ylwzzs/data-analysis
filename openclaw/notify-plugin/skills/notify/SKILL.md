# 通知发送 Skill

需要**主动发一条企微通知**时使用（不是普通对话回复）。

## 工具：send_notify

```
send_notify({ content, title?, touser?, msgtype? })
```

| 参数 | 必填 | 说明 |
|---|---|---|
| `content` | 是 | 消息正文（markdown）。直接写要通知的事，别铺垫。 |
| `title` | 否 | 标题，默认「通知」。 |
| `touser` | 否 | 收件人：省略=管理员组（默认）；`"@sender"`=当前提问用户；具体 userid（如 `ZhangDuo`）=该人；`"@all"`=全员（慎用）。 |
| `msgtype` | 否 | `markdown`(默认) / `text` / `textcard`。 |

**最常用**（告警管理员组，只填 content）：
```
send_notify({ content: "乐檬销售明细采集失败：token 过期（任务=销售明细-3120）。已重试 3 次。" })
```

通知当前用户（用户说「发给我」）：
```
send_notify({ content: "你订阅的日报已生成：今日销售额 ¥12,540。", touser: "@sender" })
```

## 什么时候发

✅ **发**：
- 采集任务完成 / 失败 / token 过期等**系统事件**告警。
- 用户**明确**要求「通知/提醒某人」「发个消息给…」「完成后告诉我」。
- 定时汇报（如每日销售摘要推送）。

❌ **不发**：
- 普通数据问答的回复 —— 那是对话，**直接答**，不要发通知。
- 用户没要求、也不是系统事件的「自作主张」通知（打扰人）。
- 一轮对话里重复发多条（合并成一条）。

## 规则

- **只通知真实发生的事**。结果/事件必须来自工具返回或明确系统状态；**绝不编造**通知内容、数字、时间。
- **content 简洁**：一两句说清什么事 + 关键数字/时间。别长篇大论。
- 默认发管理员组（省略 `touser`）。用户点名某人 → 用其 userid；用户说「发给我」→ `"@sender"`。
- 工具返回 `ok:true` 即送达，回答用户「已通知 X」；返回 `error` → 按原文告知（如密钥未配/服务不可达/无身份），**别假装已发送**。
- `@all` 广播影响全员，非用户明确要求**不要**用。

## 1. 目标

建语义层核心引擎：**视图生成器**——读 A1 的 registry + 维度，自动产出下钻视图（替代 Phase 1 那种 180 行手写 UNION ALL），配 rollup + 双轨对账保证可信。

A2 交付：生成器（通用）+ 数据源映射 + 配置清单 + 首个生成视图（门店销售下钻）+ 对账。后续视图（品类、TOP20 等）通过加清单项生成。

### 非目标（YAGNI）
- 运行时动态查询（产物是静态 PG 视图）
- 商品/客户维度（需明细，放 A4）
- admin 可视化（A3）

---

## 2. 四个已确认决策

1. **数据源**：生成器从 `report_daily_*` 聚合表（PG 可查）取数；明细 parquet 留 A4
2. **数据源映射**：新建 `metric_sources` 表（指标→聚合表+列），与 registry 口径声明解耦
3. **生成器形态**：Node 脚本读清单 + registry，**生成静态 .sql 迁移文件**，走 GHA 部署（可 review、幂等）
4. **对账**：生成器自动产 `_audit` 视图（rollup 自校验，长期）+ 独立 reconcile 脚本（双轨，迁移期临时）

## 4. metric_sources 表（数据源映射）

```sql
CREATE TABLE metric_sources (
  metric_code   TEXT NOT NULL REFERENCES metric_registry(metric_code) ON DELETE CASCADE,
  source_table  TEXT NOT NULL,        -- 聚合 PG 表（report_daily_sales 等）
  source_column TEXT,                 -- 聚合列（base 用；derived 留空）
  source_filter TEXT,                 -- 可选过滤（如 system_book_code='64188'）
  note          TEXT,
  PRIMARY KEY (metric_code)
);
```

**种子**（覆盖 A1 的 9 指标）：

| metric_code | source_table | source_column | 说明 |
|---|---|---|---|
| sale_amount | report_daily_sales | total_sale | |
| sale_profit | report_daily_sales | total_profit | 成本敏感 |
| delivery_amount | report_daily_delivery | out_money | |
| delivery_profit | report_daily_delivery | profit_money | 成本敏感 |
| wholesale_amount | report_daily_wholesale | wholesale_money | |
| wholesale_profit | report_daily_wholesale | wholesale_profit | 成本敏感 |
| outbound_amount | (derived) | NULL | 生成器按 formula 合并 delivery+wholesale 两源 |
| outbound_profit | (derived) | NULL | 同上 |
| margin | (derived) | NULL | 生成器重算 profit/amount |

derived 指标的 `source_column=NULL`，生成器按 `metric_registry.formula + depends_on` 重算。

## 5. view_manifest.yaml（生成清单）

配置文件，声明"生成哪些视图"。进 git，可 review。A3 admin 后可迁表。

```yaml
# scripts/view-manifest.yaml
views:
  - name: store_sales_drill      # → 视图 report_store_sales_drill_v
    metrics: [sale_amount, sale_profit, margin]
    dimension: branch
    levels: [region, sub_region, store]   # 用 dimension_levels 的 level_code

```

## 6. 生成器脚本 generate-views.js

**位置**：`scripts/generate-views.js`（Node，复用 services/ 的 pg 连接模式）

**依赖**：`pg`（连本地 dev PG 读 registry/sources/levels）、`js-yaml`（读 manifest）

**流程**：
1. 读 `view-manifest.yaml`
2. 连 PG，读 `metric_registry` + `metric_sources` + `dimension_levels` + `dimensions`
3. 对每个 manifest 项，按生成规则拼视图 DDL
4. 写到 `database/migrations/0NN_generated_<name>.sql`（含视图 + audit）
5. 打印生成的文件路径，开发者提交走 GHA

### 生成规则（核心算法）

对 manifest 的每项（指标集 × 维度 × 层级集），产出视图 = **每层一个 UNION ALL 分支**：

```
对每个 level（如 branch 的 region/sub_region/store）：
  SELECT
    '<level>' AS level,
    <level.key_column> AS code,
    <level.name_column> AS name,
    <parent_level.key_column> AS parent_code,      -- 串联层级
    <target_id 如果 target_scoped>,
    SUM(<source_column>) AS <metric_code>,          -- 每个 base 指标一列
    ...
    SUM(profit)/NULLIF(SUM(amount),0) AS margin     -- derived 比率重算（additive=false）
  FROM <source_table> s
  JOIN dim_branch dim ON s.branch_num = dim.branch_num   -- 维度 JOIN
  [JOIN targets t ON ... WHERE t.status='active']        -- target_scoped 时
  WHERE <source_filter> [AND is_assessed_war_zone(...)]  -- assessed_filter 时
  GROUP BY <level.key_column>, ...
```

**关键处理**：
- **derived 指标**：outbound = 合并 delivery+wholesale 两源（UNION 后 SUM，或两个 FROM 子查询）；margin = `SUM(profit)/NULLIF(SUM(amount),0)`（不直接 SUM 比率）
- **additive=false**：生成器读 registry 的 additive 标记，对比率强制重算分量
- **cost_sensitive**：生成器在视图套 `can_see_cost` CASE 脱敏（继承 security_invoker）
- **多层排序**：按 depth 排序（region 先，store 后）

### 产出文件示例

`database/migrations/080_generated_store_sales_drill_v.sql`：
```sql
-- AUTO-GENERATED by scripts/generate-views.js（勿手改，改 view-manifest.yaml 重生成）
DROP VIEW IF EXISTS report_store_sales_drill_v;
CREATE VIEW report_store_sales_drill_v AS
  -- region 层
  SELECT 'region' AS level, NULL AS parent_code, t.id AS target_id,
         dim.first_level_region AS code, dim.first_level_region AS name,
         SUM(s.total_sale) AS sale_amount, SUM(s.total_profit) AS sale_profit,
         SUM(s.total_profit)/NULLIF(SUM(s.total_sale),0) AS margin
  FROM report_daily_sales s
  JOIN dim_branch dim ON s.branch_num=dim.branch_num
  JOIN targets t ON s.system_book_code=t.system_book_code
       AND s.biz_date BETWEEN t.start_date AND t.end_date
  WHERE s.system_book_code='64188' AND is_assessed_war_zone(dim.first_level_region)
       AND t.status='active'
  GROUP BY t.id, dim.first_level_region
  UNION ALL
  -- sub_region 层（同结构，GROUP BY second_level_region，parent_code=first_level_region）
  ...
  UNION ALL
  -- store 层（GROUP BY branch_num，parent_code=second_level_region）
  ...;
ALTER VIEW ... SET (security_invoker=true); GRANT SELECT ... TO authenticated, anon;
-- + audit 视图（见 §7）
```

## 7. audit 视图（rollup 自校验，生成器自动产）

对每个生成的视图，配套 `report_<name>_v_audit`：

```sql
DROP VIEW IF EXISTS report_store_sales_drill_v_audit;
CREATE VIEW report_store_sales_drill_v_audit AS
WITH r AS (SELECT target_id, level, SUM(sale_amount) AS amt FROM report_store_sales_drill_v GROUP BY target_id, level)
SELECT target_id,
  MAX(CASE WHEN level='region' THEN amt END)     AS region_total,
  MAX(CASE WHEN level='sub_region' THEN amt END) AS sub_region_total,
  MAX(CASE WHEN level='store' THEN amt END)      AS store_total,
  ABS(region_total - sub_region_total) AS region_vs_sub_diff,
  ABS(region_total - store_total)      AS region_vs_store_diff
FROM r GROUP BY target_id;
-- 期望：所有 diff = 0（各层加总一致）
```

audit 视图**长期常驻**，可接监控（diff≠0 告警）。security_invoker + GRANT。

## 8. reconcile 脚本（双轨对账，迁移期）

**位置**：`scripts/reconcile-phase1.sql`（psql 跑，一次性）

对比新生成视图 vs Phase 1 手写视图（同 target_id、同层级）：

```sql
-- store 层销售总额：新（生成器）vs 旧（手写 report_region_breakdown_v）
SELECT
  (SELECT SUM(sale_actual) FROM report_region_breakdown_v WHERE target_id=22 AND level='store') AS old_total,
  (SELECT SUM(sale_amount) FROM report_store_sales_drill_v WHERE target_id=22 AND level='store') AS new_total,
  ABS(old_total - new_total) AS diff;
-- diff=0 → 口径一致，可下线旧视图
```

**判定**：每个 target_id、每层 diff < 1元（浮点容差）→ 通过。通过后才允许 DROP 旧手写视图。

## 11. 现状约束（雷区）

1. 视图 `DROP VIEW IF EXISTS + CREATE VIEW`，禁 `CREATE OR REPLACE`
2. 所有 DDL 幂等，迁移全量重跑
3. 加视图后须 `docker compose restart postgrest` 刷 schema 缓存（GHA 不保证）
4. 视图 `security_invoker=true` + GRANT to authenticated, anon（继承 A1 模式）
5. 明细 parquet 查不了——生成器只从 PG 聚合表取数
6. derived 比率（margin）必须 `SUM(分量)/SUM(分量)` 重算，靠 `additive=false` 保证
7. generated_*.sql 是机器产物，**勿手改**（改 manifest 重生成）
